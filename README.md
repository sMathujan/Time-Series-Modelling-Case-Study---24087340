# Time-Series Modelling Case Study — Forecasting German Electricity Demand — 24087340

Forecasting weekly German electricity demand and comparing the performance,
interpretability, and complexity of forecasting approaches — from simple
statistical benchmarks through SARIMA/SARIMAX and a feature-based ML model to an
LSTM on hourly data.

Data: hourly load from [Open Power System Data](https://data.open-power-system-data.org/time_series/)
(`DE`, Jan 2015 – Oct 2020), aggregated to weekly average load (MW → GW).
Temperature covariates come from the [Open-Meteo archive API](https://open-meteo.com/)
(Berlin, representative of German temperature).

## Repository layout

The **notebooks** are the exploratory narrative — one per assignment part, with
figures and results. The **`src/` package** distils the same logic into reusable,
tested functions, and **`scripts/run_pipeline.py`** reproduces the weekly model
comparison in one command.

```text
├── 01_Data_Loading_and_EDA.ipynb          # Part 1: load, aggregate, EDA, stationarity
├── 02_Benchmark_Forecasting.ipynb         # Part 2: mean / naive / seasonal naive / drift
├── 03_SARIMA_Modelling.ipynb              # Part 3: SARIMA grid search + diagnostics
├── 04_SARIMAX_Exogenous_Regressor.ipynb   # Part 4: + temperature (conditional forecast)
├── 05_Feature_Model.ipynb                 # Part 5: Random Forest / Gradient Boosting
├── 06_LSTM.ipynb                          # Part 6: LSTM on hourly data
│
├── src/electricity_demand/
│   ├── config.py        # paths and constants
│   ├── data.py          # OPSD load + Open-Meteo temperature
│   ├── features.py      # calendar / temperature / leakage-safe supervised table
│   ├── evaluation.py    # MAE / RMSE / MASE / Bias, interval coverage
│   ├── plotting.py      # forecast + error-diagnostic figures
│   ├── pipeline.py      # end-to-end weekly pipeline
│   └── models/          # benchmarks, sarimax, feature_models, neural (LSTM), bayesian
│
├── scripts/             # download_data, make_features, run_pipeline, evaluate_models
├── tests/               # pytest: benchmarks, evaluation, feature-leakage
├── data/processed/      # small processed CSVs (regenerated by scripts)
├── outputs/             # figures / forecasts / metrics (regenerated)
├── requirements.txt · environment.yml · pyproject.toml · LICENSE (Apache-2.0)
```

## Installation

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

`tensorflow` is only needed for the LSTM (`06_LSTM.ipynb` / `models/neural.py`);
the rest of the pipeline runs without it. Conda users: `conda env create -f environment.yml`.

## Reproducing the analysis

Two ways, sharing the same underlying code:

**Notebooks** — run `01` → `06` in order; each saves its processed data and
figures.

**Pipeline (one command)** — reproduces the weekly benchmark/SARIMAX/feature
comparison:

```bash
python scripts/download_data.py     # OPSD hourly load + Berlin temperature
python scripts/make_features.py     # build data/processed/feature_df.csv + weekly_load.csv
python scripts/run_pipeline.py      # fit models, evaluate, save outputs
```

Outputs:

```text
outputs/forecasts/all_forecasts.csv
outputs/metrics/model_comparison.csv
outputs/figures/forecast_comparison.png
outputs/figures/error_diagnostics.png
```

The LSTM stays in `06_LSTM.ipynb` by default (it needs the hourly series and
TensorFlow); enable it in the pipeline with `include_neural=True`.

## Models & evaluation

All models are scored on the same final **104 weeks** (2-year horizon) with
**MAE, RMSE, MASE, Bias**; MASE is scaled by the in-sample seasonal-naive error,
making seasonal naive the reference benchmark. The split is strictly
chronological.
